# autable 权限模型

本文档是 autable 权限系统的唯一权威规范。所有权限相关的实现、评审、排查都以本文为准;实现与本文不一致时,以本文为目标修正实现。文末的[实施状态](#实施状态)记录当前落地进度。

## 设计原则

1. **默认拒绝(default deny)**:没有授权就没有权限。任何"未被权限覆盖所以放行"的行为都是漏洞。
2. **服务端是唯一边界**:前端的一切过滤只是体验优化。任何用户直接调用任何 API 的组合,都不能看到或改到授权之外的数据。
3. **两层分离**:作用于**数据**(cell 值、行)的权限与作用于 **metadata**(schema)的权限是两套独立的东西,互不隐含。
4. **权限的边界本身只能由 owner 定义**:能改变"别人能看到什么"的操作(视图定义、公式、授权、角色)全部 owner 独占。可授权出去的只有"在既定边界内做事"的能力。

## 主体(Subject)

| 主体 | 标识 | 说明 |
|---|---|---|
| 用户 | user id | 登录用户 |
| 角色 | `role:<db>:<name>` | 一组授权的集合;成员(用户或 workflow)继承角色的全部授权,取并集 |
| workflow | `workflow:<id>` | workflow 内的表操作节点以此身份执行,权限由 owner 配置 |

有效权限 = 直接授权 ∪ 所属各角色的授权,同一权限取最高档。数据库 **owner 绕过一切检查**(published form 的显式授权是唯一例外)。

## 数据层权限

### 字段位权限(读 / 改 / 增)

每个字段三个**互相独立**的开关,存储为位掩码:

| 位 | 含义 | 生效时机 |
|---|---|---|
| 读 read | 返回的行里包含该字段的值 | 一切读取路径(列表、查询、写操作的响应体、历史) |
| 改 update | 允许修改已存在行的该字段 | `UpdateRow` / upsert 命中已有行 |
| 增 create | 允许在新建行时填写该字段 | `CreateRow` / upsert 新建行 |

- 三者独立:可以配"只能建时填、之后不能改"(一次性字段),也可以配"能改但读不回显"。
- 无读权限的字段:值在所有响应体中**一律剔除**——包括创建/修改/删除操作自己的响应体。字段的存在(名字、类型)不保密,值保密。
- 写请求中出现无权字段 → 整个请求 403 拒绝(不做静默丢弃)。
- 兼容:历史授权 Level `1` 解释为 `读`,`2` 解释为 `读|改|增`。
- `field_set` 授权是对整表所有字段的批量位授权,与单字段授权取并集。

### 行级权限:视图 = 行集边界

**视图是 record 层权限的唯一边界。** 一个主体能接触到的行(行集)= 其拥有**视图读**授权的所有视图所筛选行的**并集**。

- **"all" 是一个真实的内置视图**(无过滤条件,即全表),和其他视图一样需要授权才能读。API 中视图名为空一律按 `all` 解释——不存在"不带视图 = 无限制"的路径。
- 读取必须指定一个已授权的视图;请求级的 query / search / sort 只能在视图结果上 **AND 收窄**,永远不能放宽。
- 视图查询支持 **`$current_user` 动态变量**,解析时替换为**当前用户的 email**。一个 `提交人 = $current_user` 的视图即可实现按人隔离,不需要每人建一个视图。非用户主体(workflow、角色本身)没有 email,`$current_user` 对它们解析为其主体标识,通常不会命中任何行——给 workflow 授权时应选择不含 `$current_user` 的视图。
- **视图的新建、修改、删除 owner 独占**。视图定义就是行级权限的定义,能改视图 = 能改权限边界。`view_set` / `view` 授权只表达"可以读哪些视图"。

### 写入规则

| 操作 | 行级校验 | 其他要求 |
|---|---|---|
| 新建行 | **无**(行还不存在,校验无意义) | `record.create` + 所填字段的`增`位 |
| 修改行 | **必须**:目标行 ∈ 行集,否则 404 | 所改字段的`改`位 |
| 删除行 | **必须**:目标行 ∈ 行集,否则 404 | `record.delete` |
| upsert | match 查询**限定在行集内**;命中→按修改处理,未命中→按新建处理 | 对应上两行 |

- 新建永远明确返回成功/失败(表单场景用户必须得到确定的提交结果),即使新建出的行落在自己行集之外、随后自己看不见。
- **凡是触碰已存在行的操作,一律行级校验**。行不在行集内时返回 404(按不存在处理),不返回 403——不泄露行的存在性。
- `record.create` / `record.delete` 是表级的行操作开关,与字段位权限相互独立、同时校验。

### 文件

- 文件上传后处于未绑定状态,**任何人不可见**;由上传者本人一次性绑定到某行(不可移动到其他表/行)。
- 文件的下载与元数据读取是**行级**的:文件所绑定的行必须在访问者的行集内,且访问者拥有该表的 `file` 授权。表级 file 授权不再单独构成可见性。

### 关联字段(relation)

- 关联字段存储的只是目标表的 record id,本身不泄露数据。
- 关联标签(id → 显示文本)的解析受目标表行集约束:目标行不在访问者行集内时只显示 `#id`。需要展示标签时,由 owner 给目标表授一个合适的只读视图。

## metadata 层权限

作用于 schema,与数据层权限完全独立(有 `field_add` 不代表能读写任何数据,反之亦然)。

| 权限 | 粒度 | 允许的操作 | 典型授权对象 |
|---|---|---|---|
| `field_add` | 表 | 新增**非公式**字段 | sync 类 workflow(ensure_fields 自动建列) |
| `field_modify` | 字段 | 修改**该**字段的定义:类型(string/int/float)、relation 目标等,**不含公式** | 需要调整表结构的协作者 |

**owner 独占清单**(不可授权):

- 公式字段的一切:新建公式字段、修改公式体、把字段改成公式。公式在服务端以**全行数据**(含定义者之外任何人不可读的字段)求值,是唯一能横穿字段读权限的机制,因此只能由本就看得到一切的 owner 定义。
- 删除字段、删除视图、视图的增改(见上)。
- 表的创建与删除、数据库管理。
- 授权(grants)与角色(roles)的一切管理。

## API 行为规范

所有返回行数据的端点必须遵守:

1. **读取**(`GET /rows`、`POST /rows/query`、`/rows/page`、搜索):必须解析出一个已授权视图(空 = `all`),`CanReadView` 不过 → 403;结果行 ∈ 该视图,值经字段读过滤;`total` 只统计视图内的行。
2. **写入**(`POST /rows`、`PATCH /rows/{id}`、`DELETE /rows/{id}`、`/rows/upsert`):按[写入规则](#写入规则);**响应体的行值必须经过字段读过滤**。
3. **行历史**(`GET /rows/{id}/history`):目前 owner-only。放开的前提:目标行 ∈ 行集 **且** 历史值经字段读过滤,二者缺一不可。
4. **文件**(`GET /files/{id}`、`POST /files/metadata`):行级校验(见上)。
5. 查询/排序/搜索中引用无读权限的字段 → 403(防侧信道);全文搜索只在可读字段上构建。
6. workflow 表节点(`table.row.*`、`table.field.create`)以 `workflow:<id>` 为主体走完全相同的检查,无任何捷径。

## 已知的非强制边界(文档约定,不做机器强制)

这些是刻意的设计取舍,授权时需要知晓:

- **公式字段是 owner 定义的投影**:公式可读全行。给某人某公式字段的读权限,等于让他看到公式所表达的任何信息。owner 写公式时自行负责。
- **workflow 运行历史含数据**:run 的输入/输出/步骤会包含它处理过的行数据,且不做字段过滤。授权某人读某 workflow ≈ 授权他看该 workflow 处理的数据。
- 字段名、表名、视图名等 schema 信息对有该表任意授权的用户可见,不保密。

## 升级与迁移

行级边界(内置 `all` 视图需授权)上线采用**一刀切默认拒绝**:升级后非 owner 主体在获得视图授权前读不到任何行。owner 按需授权视图(通常从 `view:all read` 或一个 `$current_user` 隔离视图开始)。不做自动补发,换取模型从第一天起就是干净的默认拒绝。
